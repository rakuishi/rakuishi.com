# ブログ執筆スタイルガイド

## フロントマター構造

```yaml
---
category: tech | blog
date: "2025-01-01T10:00:00+09:00"  # ISO 8601、タイムゾーンは +09:00
page: false                        # 固定ページの場合のみ true
slug: article-slug-in-kebab-case
title: "記事タイトル"
---
```

`category` は `tech`（技術記事）か `blog`（日記・エッセイ）の二択。

## 共通の文章の傾向

### 半角英数字の前後に半角スペースを付与する

日本語と半角英数字（英語・数字）が隣り合う場合は、必ず半角スペースで区切る。

- iOS アプリを Swift に移行しました
- AI の支援を受けながら Objective-C から Swift に移行しました
- Xcode 16 以降は、メニューから簡単に変換できるようになってます
- 同じ PR でのテスト時間は 12m → 3m に減少し
- LLM の仕組みと似ている

### 約物と英数字の間にはスペースを入れない

句読点（。、）・括弧（（））・カギ括弧（「」）などの約物は、それ自体が境界として機能する。直後・直前に英数字が来る場合でも、余分なスペースは入れない。

- ようになってます。Xcode 上の見え方と、実際のフォルダ上の見え方を揃えた上で（。と X の間にスペースなし）
- さらに、Swift のクラス側に `@objcMembers` を追加します（、と S の間にスペースなし）
- が、LSD の幻覚の働きは次のように説明される（、と L の間にスペースなし）
- Kotlin スクリプト（KTS）が推奨されるため（（ の前、）の後ともにスペースなし）
- 大規模言語モデル（LLM）の仕組みと似ている（同上）

### 書名・作品名は『』で囲む

- 『利己的な遺伝子』
- 『脳は世界をどう見ているのか　知能の謎を解く「1000 の脳」理論』
- 『神は妄想である―宗教との決別』
- 『幻覚剤は役に立つのか』

### 補足・注釈には全角括弧（）を使う

文中の補足や読者向けの脱線は、全角括弧で差し込む。半角括弧は使わない。

- （ある亀は肛門呼吸を行え、浮上する必要さえないそうだが、それは置いておく）
- （もちろん SNS は緩い関係が続くため、SNS がない時代に比べて途絶えにくいとは思うが）
- （と想像しているが、もっと複雑な祈りの手順があるのかもしれない）

### 数字は半角を使う

- Xcode 16 以降は
- 2 時間で駄目になり
- 定量的な値になおせば、短期的な関係を求めるドーパミンは 2/3 程度、長期的な関係を求めるオキシトシンは 1/3 程度だろうか

### 絵文字は使わない

いかなる状況でも絵文字は使用しない。

### 避けるべき表現

以下の表現は著者の文体に合わないため使わない。

冗長な動詞表現は短くする。

- 〜することができます → 〜できます
- 〜を行う → 〜する
- 〜となっています → 〜です / 〜だ

定型の締め・クッション表現は使わない。

- 参考になれば幸いです、お役に立てれば幸いです
- 最後までお読みいただきありがとうございました
- 以上となります、〜させていただきます

根拠のない曖昧な断言・過度な修飾も避ける。

- 非常に・様々な・多くの を多用しない
- 〜と思われます・〜かもしれません を多用しない（著者が使う場合は、真に不確かさを表す文脈のみ）
- blog 記事で不要な見出しや箇条書きを挿入しない（著者は散文で書く）

## カテゴリ別 文体ガイド

### blog カテゴリ（エッセイ・日記）

#### 構造

- 見出し（`##`）は原則として使わない。流れるような散文で書く
- 段落は短め。1 段落に長い考察を詰め込みすぎない
- 結論は開かれたまま終わることが多い。断定せず、思考の過程を見せる

#### 文体

- 一人称は私・自分・僕が混在する（統一しなくてよい）
- 体験・エピソードから始まり、そこから抽象的・哲学的な考察へと展開する
- 科学的な知見（進化論、神経科学、認知科学など）を引用して、個人的な感情や疑問と接続する
- 懐疑的でニヒルな視点を持ちながらも、最終的には静かな受容や諦観で着地する
- ユーモアは淡く、括弧書きで差し込まれることが多い
- 逆説的な構造を好む（`～にもかかわらず` `しかし` `が、`）
- 比喩表現を使う。具体的で視覚的なものが多い
  - セメントを産出するために切り崩される山のように
  - 月に一度は不安という獣に食い散らかされ
  - 足もとに転がっている石ころの形や色が綺麗だなと思うこともあって

#### よく使うパターン

- `だが` `しかし` `が、` による逆接転換
- `そう考えると` `結局のところ` による収束
- 定量的な表現で感覚を表す（2/3 程度、1/3 程度）
- 読んだ本への言及と個人的な感想の組み合わせ
- 読点でつなげて長い思考の流れを作ることがある

#### 書き出しのパターン

具体的な光景・情景・体験から始め、感覚に訴えるものから入る。

- 大きなガラス窓の外から、軟球テニスのぱこんと打ち返す音が響く。
- 帰省するたびに祖母はあなたのためを思って祈っているのだからと言う。
- 亀のように内向的な私であるが、たまには息を吸うために浮上して外交的に振る舞うことがある。

#### 書き終わりのパターン

断定せず、静かな受容・諦め・余韻で締める。開かれた問いや継続する思考を示唆することもある。

- 結局は、そういう素直なままを受けとっていくしかないのかなと思っている。
- 今は冬、冷たい手で読んだ本を思い出す。
- そのような浮き沈みを繰り返しながら藻掻いている……。

### tech カテゴリ（技術記事）

#### 構造

- `##`（h2）から始まるセクション見出しで明確に構成する
- 必要に応じて `###`（h3）でサブセクションを作る
- 参考リンクは `参考：[タイトル](URL)` の形式でセクション末尾に置く
- コードブロックにはファイル名を `title` 属性で付ける（shiki-transformer-filename 対応）

```
```swift title="ViewController.swift"
```

#### 文体

- 客観的で実務的なトーンだが、硬すぎない
- 作業手順は番号付きリスト（`1.` `2.` `3.`）で整理する
- 自分が実際にやった体験として書く（自分は〜しました、〜することにしました）
- 判断の背景も簡潔に記す（この順番で作業するのが最適なのかわかりませんが、など正直に書く）
- AI ツールの活用など時代の文脈も自然に織り交ぜる

#### 書き出しのパターン

何をしたかを 1〜2 文で端的に伝え、なぜ書いたかの動機も添える。

- 個人プロジェクトの iOS アプリは Objective-C で記述されていましたが、今回、AI の支援を受けながら Swift に移行しました。
- 最新の Android Studio から新規プロジェクトを作成すると、見慣れない書き方がちらほら登場します。この記事では、仕事の Android プロジェクトをその最新のビルド構成に追従したときの覚え書きと...について触れます。
- 静的サイトジェネレーター Astro で運用しているブログの環境整備を行いました。

#### 書き終わりのパターン

作業完了を宣言して締める。今後の展望を 1 文添えることもある。

- 以上で移行完了しました。お疲れ様でした！
- 来年はリリース作業も自動化したいと思っています。
